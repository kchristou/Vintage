Description of Testing:

The product that we want to test in this part is the treasure hunt which is a web application and it is targeting mobile devices first and then Desktop Screens. It requests data from a server and it implements back a few functions (e.g./list , /leaderboard etc).
• When the user will do the testing?
When we give the the web-app for testing.

• What happens when a defect (bug) is discovered?
When the users find any problem and mention it to us through our email, social media or even by telling to us face to face one of the team members is going to fix it.

User Acceptance Plan:

The application was given to members of the other teams and our class tudors in order for us to have a black box testing which will help us to find and identify mistakes.
Some issues that they found:
♦ Images Bigger on Computer Screens
♦ Some buttons were big and some others small
♦ The Link of social media was not linked to the right pages

Usability Assessment

Heuristic: How: Done/Not Done:
1: Visibility of system status Displaying all Errors ✔
2: Match between system and the real world Use icons like pirates and Old maps like the real world ✔
3: User control and freedom User can choose if he/she is going to open the camera and location ✔
4: Consistency and standards The design of all pages is the same. (Have same consistency) ✔
5: Recognition rather than recall Use of QR Scanner in order to answer some questions ✔
6: Flexibility and efficiency of use When a user refresh the page the game continues with the questions that are left ✔
7: Aesthetic and minimalist design Design based on Pirates and treasure hunts of Cartoons ✔
8: Help users recognize, diagnose, and recover from errors Warnings if some questions are answered wrong ✔
9: Help and documentation The functionality of the app is explained through help page ✔
10: Error prevention Error messages to inform the users in order to prevent mistakes when they answering questions ✔
